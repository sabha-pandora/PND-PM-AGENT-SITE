What the PM Agent Does

An AI-powered product management assistant that integrates directly with any Pandora team's Jira board and Confluence workspace via REST API. Switch teams by changing one env variable.

✍

Write User Stories

Create new Jira stories from natural language descriptions. Automatically follows "As a / I want / So that" format, INVEST criteria, and links to the correct parent epic.

Jira REST API

Testable Acceptance Criteria

Generates acceptance criteria in Given/When/Then (Gherkin BDD) format. Every story gets minimum 3 scenarios: happy path, edge case, and negative path.

Gherkin BDD

Resolve Team Queries

Auto-Response

Confluence Documentation

Confluence API

Quality Framework

Built-in Best Practices

Every action the agent takes is guided by Pandora Digital's quality engineering standards.

☑ INVEST Criteria

  • Independent — self-contained, no hidden dependencies
  • Negotiable — describes what, not how
  • Valuable — delivers clear user/business value
  • Estimable — enough detail to size
  • Small — fits in one sprint (1-3 days)
  • Testable — clear pass/fail acceptance criteria

☑ Gherkin BDD Format

  • Given a precondition or context
  • When an action is performed
  • Then the expected outcome occurs
  • Minimum 3 scenarios per story
  • Happy path + edge case + negative path
  • Atomic: one behavior per scenario

☑ SPIDR Decomposition

  • Spike — research unknowns first
  • Paths — split by user workflows
  • Interfaces — split by channels
  • Data — split by variations
  • Rules — split by business rules

☑ Anti-Pattern Detection

  • Rejects vague language ("works correctly")
  • Catches solutions-in-disguise
  • Flags missing negative/error paths
  • Splits compound criteria automatically
  • Identifies assumed context

☑ Definition of Ready

  • As a / I want / So that format
  • ACs reviewed by QE
  • Sized and estimated by dev team
  • Dependencies documented
  • Mockups & API contracts attached
  • Linked to parent Epic

☑ Definition of Done

  • All ACs met and signed off by PO
  • Unit tests passing (min 80% coverage)
  • Functional tests executed & linked
  • No open Critical/High defects
  • Code peer reviewed & merged
  • Docs and release notes updated

Open Windows Terminal or Command Prompt — press Win, type Terminal or cmd, press Enter. Then run this single command to install Claude Code:

One-time Setup

Get the PM Agent onto Your Machine

Create a branch for your team and download the agent.

In Terminal, navigate to where you want to keep the project (e.g. your Desktop), then run:

In Windows Terminal / Command Prompt, navigate to where you want to keep the project (e.g. your Desktop), then run:

# Clone the repo git clone https://github.com/sabha-pandora/PND-PM-AGENT.git cd PND-PM-AGENT # Create a branch for your team (replace CARE with your team name) git checkout -b team/CARE

3
Configuration

Configure Your Jira & Confluence Credentials

Tell the agent which project board to connect to and how to authenticate.

Copy the example config file and open it in any text editor:

cp .env.example .env open .env
copy .env.example .env notepad .env

Fill in these values — replace everything inside the quotes:

# Your login email JIRA_EMAIL="your-name@pandora.com"
# From id.atlassian.com → Security → API tokens JIRA_API_TOKEN="paste-your-token-here"
# Visible in your Jira board URL JIRA_PROJECT_KEY="CARE"
# Human-readable team name PROJECT_NAME="Pandora Care Team"
# Used as prefix in story titles STORY_PREFIX="CARE"
# Confluence → Space Settings → Space Details CONFLUENCE_SPACE_ID="your-space-id"
🔑

Where is my Jira project key? Open your Jira board in a browser. Look at the URL — it will say something like jira.../projects/CARE/boards. The word in capitals is your project key.

🔒

The .env file is private. It is listed in .gitignore so it will never be accidentally committed or shared. Never paste your API token anywhere else.

First Run

Sync Your Knowledge Base

Pull your entire Jira board into a local snapshot so the agent has full context.

Run this command once when you first set up, and again whenever your board has changed significantly:

/sync-kb

This fetches all your epics, stories, acceptance criteria coverage, and test gaps into docs/care/knowledge_base.json. The agent reads this to give you relevant, board-aware answers instead of generic ones.

📅

When should I re-run it? The agent will remind you if the snapshot is more than 7 days old. As a rule of thumb: run it at the start of each sprint, after a bulk import of stories, or whenever you've added a new epic.

  • Run at the start of each sprint
  • After importing stories in bulk
  • After adding a new epic or major feature area
  • When the agent's answers seem out of date

Good to Know

Switching Between Teams

If you support more than one Pandora team, switching takes seconds.

Open your .env file (open .env in Terminal) and change the three team lines:

Open your .env file (notepad .env in Command Prompt) and change the three team lines:

JIRA_PROJECT_KEY="LOYALTY"← your team's Jira key
PROJECT_NAME="Pandora Loyalty Team"← team display name
STORY_PREFIX="LOYALTY"← prefix in story titles

Then run /sync-kb to load that team's board. Every command will now operate against the Loyalty project. Switch back by changing the values again.

📁

Each team's data is kept separate. PVA data lives in docs/pva/, CARE in docs/care/, LOYALTY in docs/loyalty/ — switching teams never overwrites another team's knowledge base.

Pro Tip

Scan Your Dev & QE Repos Before Writing Stories

See what the team has already built so you don't write stories for finished work.

If your team's code lives on your machine, point the agent at the folder:

/scan-repo /Users/yourname/repos/pva-chatbot
/scan-repo C:\repos\pva-chatbot

If your team uses Azure DevOps, add your credentials to .env first:

AZURE_ORG="pandoradigital"← your Azure DevOps org name
AZURE_PROJECT="PVA"← your Azure project
AZURE_PAT="paste-your-pat-here"← Personal Access Token
DEV_REPO_NAME="sfcc-chatbot"← your DEV repo name
QE_REPO_NAME="sfcc-qe"← your QE repo name

Then run:

/scan-repo pandoradigital/PVA/sfcc-chatbot --days=30

Or scan both DEV and QE repos together, and link the summary to a Jira epic:

/scan-repo dev:/repos/sfcc-dev,qe:/repos/sfcc-qe --epic=PVA-100

The agent will produce a plain-English summary grouped by feature area — chatbot flows, payment, auth, etc. — and save it to your team's docs folder. Next time you run /write-story, the agent automatically reads this summary for context.

🔑

Where do I get an Azure PAT? Go to dev.azure.com → click your avatar → Personal Access Tokens → New Token. Set scope to Code (Read) and Pull Request Threads (Read). It only needs read access — no write permissions required.

📅

When should I re-run it? Run at the start of each sprint or before a planning session. Use --days=14 for a quick "what changed this sprint" view, or --days=90 for a full quarterly retrospective.

  • Run before sprint planning to see what's recently been shipped
  • Use to validate stories — if it's already in git, mark it done
  • Use the QE repo scan to check which areas have automated test coverage
  • Combine with /write-story for context-aware story creation
